:::tip[Коротко]
Виртуальное окружение — изолированная папка с пакетами под один проект. Без неё библиотеки разных проектов конфликтуют по версиям. Стандартный инструмент — встроенный **`venv`**: создал, активировал, ставишь пакеты — они живут только в этом проекте.
:::

## Зачем это нужно

Проект А хочет pandas 1.5, проект Б — pandas 2.2. Если ставить пакеты глобально, они затрут друг друга. venv даёт каждому проекту свой набор пакетов — никаких конфликтов и «у меня работало».

## Создание и активация

В папке проекта:

```bash
python -m venv venv        # создать окружение в папке venv/
```

Активировать:

```bash
# macOS / Linux
source venv/bin/activate

# Windows (PowerShell)
venv\Scripts\Activate.ps1
```

После активации в начале строки появится `(venv)` — значит, ты внутри. Теперь `pip install pandas` ставит пакет только сюда. Выйти — командой `deactivate`.

## requirements.txt

Чтобы зафиксировать список пакетов и повторить окружение на другой машине:

```bash
pip freeze > requirements.txt        # сохранить текущие версии
pip install -r requirements.txt      # установить из файла
```

`requirements.txt` коммить в Git — это рецепт окружения. Саму папку `venv/` в Git **не** клади (внеси в `.gitignore`).

## venv vs conda

| | venv | conda |
|--|------|-------|
| Идёт с | Python (встроен) | дистрибутив Anaconda/Miniconda |
| Ставит | только Python-пакеты | пакеты + не-Python (C-библиотеки) |
| Когда | большинство задач аналитика | тяжёлый научный стек, проблемные сборки |

Для старта хватает `venv`. conda пригодится, если столкнёшься с пакетами, которые тяжело собираются через pip.

:::caution[Выбери окружение в редакторе]
Создал venv — укажи его интерпретатор в [VS Code](/01-setup/ide-setup/) (`Python: Select Interpreter`). Иначе ноутбук будет брать глобальный Python и «не видеть» установленные пакеты.
:::
