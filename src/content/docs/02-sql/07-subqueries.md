:::tip[Коротко]
Подзапрос — это `SELECT` внутри другого запроса. Виды:

- **Скалярный** — возвращает одно значение (можно в `SELECT` или `WHERE`).
- **`IN (SELECT ...)`** — «входит в список из подзапроса».
- **`EXISTS (SELECT ...)`** — «есть ли хоть одна строка» (часто быстрее и безопаснее `IN`).
- **Коррелированный** — ссылается на внешний запрос, выполняется для каждой строки.

Частый приём аналитика: сначала свернуть данные подзапросом, потом джойнить — чтобы убить дубли.
:::

:::note[Поток данных]
Вход: таблица(ы)
→ Обработка: внутренний запрос считает промежуточный результат → внешний использует его (в `WHERE`/`FROM`/`SELECT`, в т.ч. коррелированно)
→ Выход: итоговая таблица.
Зачем: построить ответ в два шага, когда одним плоским запросом не выразить.
:::

## Зачем это нужно

Подзапросы отвечают на вопросы «в два этапа»: сначала посчитать что-то промежуточное, потом использовать. «Заказы выше среднего», «клиенты без заказов», «доля заказа в общей выручке» — всё это подзапросы.

```sql title="Демо-данные"
-- та же схема, что и в разделе (см. Установку окружения)
INSERT INTO customers VALUES (1,'Аня','RU'),(2,'Борис','RU'),(3,'Кира','KZ'),(4,'Лев','DE');
INSERT INTO orders VALUES
    (101,1,'paid',2500),(102,1,'paid',1800),
    (103,2,'cancelled',990),(104,3,'paid',4200),(105,9,'paid',700);
```

## Скалярный подзапрос

Возвращает ровно одно значение. Удобно сравнить каждую строку с агрегатом по всей таблице:

```sql
-- заказы выше среднего чека
SELECT order_id, amount
FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);
```

Средний `amount` = (2500+1800+990+4200+700)/5 = 2038. Выше него:

| order_id | amount |
|----------|--------|
| 101      | 2500   |
| 104      | 4200   |

Скалярный подзапрос можно положить и в `SELECT` — например, посчитать долю заказа в оплаченной выручке (подзапрос и снаружи, и внутри считает по `paid`, поэтому доли складываются в ~100%):

```sql
SELECT order_id,
       amount,
       ROUND(amount * 100.0 / (SELECT SUM(amount) FROM orders WHERE status = 'paid')) AS pct
FROM orders
WHERE status = 'paid';
```

Знаменатель — `SUM(amount)` по оплаченным = 9200.

| order_id | amount | pct |
|----------|--------|-----|
| 101      | 2500   | 27  |
| 102      | 1800   | 20  |
| 104      | 4200   | 46  |
| 105      | 700    | 8   |

## IN с подзапросом

`IN (SELECT ...)` — отфильтровать по списку, который сам считается запросом:

```sql
-- клиенты, у которых есть оплаченные заказы
SELECT name
FROM customers
WHERE customer_id IN (SELECT customer_id FROM orders WHERE status = 'paid');
```

| name |
|------|
| Аня  |
| Кира |

:::caution[NOT IN + NULL = пусто]
Если подзапрос в `NOT IN` вернёт хоть один `NULL`, весь результат станет пустым (см. [Операторы фильтрации](/02-sql/04-filtering-operators/)). Поэтому для «найти без пары» надёжнее `NOT EXISTS`.
:::

## EXISTS / NOT EXISTS

`EXISTS` проверяет, **есть ли хоть одна строка** в подзапросе — не важно, какая. Часто быстрее `IN` (СУБД останавливается на первом совпадении) и не ломается на `NULL`.

```sql
-- клиенты БЕЗ единого заказа
SELECT name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);
```

| name |
|------|
| Лев  |

Внутри `EXISTS` пишут `SELECT 1` — конкретные столбцы не нужны, проверяется лишь факт наличия строки.

## Коррелированный подзапрос

Это подзапрос, который **ссылается на внешний** запрос (`o.customer_id = c.customer_id` выше) и выполняется заново для каждой внешней строки. Мощно, но медленно на больших данных — по сути цикл.

```sql
-- последний (максимальный по id) заказ каждого клиента
SELECT c.name,
       (SELECT MAX(o.order_id) FROM orders o WHERE o.customer_id = c.customer_id) AS last_order
FROM customers c;
```

| name  | last_order |
|-------|------------|
| Аня   | 102        |
| Борис | 103        |
| Кира  | 104        |
| Лев   | NULL       |

:::note[Коррелированный подзапрос ≈ оконная функция]
То же самое часто элегантнее и быстрее решается [оконными функциями](/02-sql/09-window-functions/) (`MAX() OVER (PARTITION BY ...)`). Если видишь коррелированный подзапрос в `SELECT` — подумай про окно.
:::

## Подзапрос против JOIN: чинит дубли

Главное прикладное применение: свернуть «многие» в подзапросе **до** соединения, чтобы не задвоить суммы (вспомни fan-out из [JOIN-ов](/02-sql/06-joins/)).

```sql
-- ✅ сначала агрегируем order_items по заказу, потом джойним — без дублей
SELECT o.order_id, o.amount, items.cnt
FROM orders o
JOIN (
    SELECT order_id, SUM(qty) AS cnt
    FROM order_items
    GROUP BY order_id
) AS items ON items.order_id = o.order_id;
```

Такой подзапрос в `FROM` называют **производной таблицей**. Когда их становится много или они вложены — переходи на [CTE](/02-sql/08-cte/) (`WITH`), читается в разы лучше.

<details>
<summary>1. Клиенты, чей суммарный чек выше, чем у клиента №2.</summary>

```sql
SELECT customer_id, SUM(amount) AS total
FROM orders
GROUP BY customer_id
HAVING SUM(amount) > (
    SELECT SUM(amount) FROM orders WHERE customer_id = 2
);
```

Скалярный подзапрос считает порог (сумму клиента 2 = 990), `HAVING` сравнивает с ним.

</details>

<details>
<summary>2. Товары, которые ни разу не заказывали.</summary>

```sql
SELECT title
FROM products p
WHERE NOT EXISTS (
    SELECT 1 FROM order_items oi WHERE oi.product_id = p.product_id
);
```

`NOT EXISTS` безопаснее `NOT IN` — не сломается, если в `order_items` встретится `NULL`.

</details>

<details>
<summary>3. Почему EXISTS часто быстрее IN?</summary>

`EXISTS` останавливается на первой найденной строке (нужен лишь факт наличия), тогда как `IN` может материализовать весь список значений подзапроса. Плюс `EXISTS`/`NOT EXISTS` корректно работают с `NULL`, а `NOT IN` — нет.

</details>
