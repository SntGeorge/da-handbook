:::tip[Коротко]
Отчёт из Desktop публикуется в **Power BI Service** (облако), где живёт в **workspace** и раздаётся через **apps**. Данные обновляются по расписанию (**scheduled refresh**). Для разграничения доступа к строкам — **Row-Level Security (RLS)**: один отчёт, но каждый видит только свои данные.
:::

## Зачем это нужно

Отчёт в `.pbix` на твоём ноутбуке бесполезен для команды. Публикация в Service делает его доступным, обновляемым и безопасным. Понимание refresh и RLS — обязательный минимум для работы в команде.

## Power BI Service

**Publish** из Desktop → отчёт загружается в облако Power BI Service (app.powerbi.com). Там его смотрят в браузере и на мобильном, без установки Desktop.

## Workspace и Apps

- **Workspace** — рабочее пространство команды: отчёты, датасеты, дашборды. Здесь идёт разработка и совместная работа.
- **App** — «опубликованная витрина» из workspace для конечных пользователей: чистый, стабильный набор отчётов с управляемым доступом. Команда правит в workspace, потребители смотрят app.

## Refresh данных

| Тип | Что делает |
|-----|-----------|
| **Scheduled refresh** | обновление по расписанию (до 8 раз/день на Pro) |
| **Incremental refresh** | догружает только новые/изменённые данные — для больших таблиц |

:::note[Для refresh из локального источника нужен Gateway]
Если данные лежат не в облаке, а в локальной БД/файле, Service не достанет их сам — нужен **On-premises Data Gateway** (мостик между облаком и твоей сетью). Без него запланированное обновление будет падать. Для облачных источников gateway не нужен.
:::

## Row-Level Security (RLS)

RLS ограничивает доступ к **строкам** по пользователю: менеджер РФ видит только российские данные в том же отчёте. Настраивается ролями с DAX-фильтром:

```dax
[Country] = "RU"            -- роль "РФ-менеджер" видит только эти строки
```

Роли создаются в Desktop (Manage roles), а пользователи назначаются в Service. Это стандартный способ дать один отчёт многим с разной видимостью данных.

## Шаринг

Поделиться можно: добавив в workspace, опубликовав app, отправив ссылку (share) или встроив (embed). Для большинства внутренних сценариев правильный путь — **app** с настроенными правами, а не рассылка `.pbix`-файлов.

## Задачи для самопроверки

<details>
<summary>1. Запланированное обновление отчёта из локальной PostgreSQL падает. Почему?</summary>

Power BI Service не имеет прямого доступа к локальной БД за пределами облака. Нужен On-premises Data Gateway — мостик, через который Service дотягивается до локального источника. Для облачных источников (BigQuery, SQL в облаке) gateway не требуется.

</details>

<details>
<summary>2. Нужно, чтобы менеджеры регионов видели один отчёт, но каждый — только свои данные. Чем?</summary>

Row-Level Security (RLS): создать роли с DAX-фильтром по региону (`[Region] = "..."`) в Desktop и назначить пользователей в Service. Отчёт один, но строки фильтруются под каждого. Это та же идея, что [RLS в Tableau](/07-bi-tools/tableau/07-publishing/).

</details>
