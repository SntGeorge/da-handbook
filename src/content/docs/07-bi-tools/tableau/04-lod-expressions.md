:::tip[Коротко]
LOD-выражения (Level of Detail) задают **свой уровень детализации** для расчёта, независимый от того, что на полках. Главное — **`FIXED`**: считает агрегат по указанным измерениям, игнорируя остальную разбивку. Нужны для задач вроде «доля клиента в общей выручке» или «средний чек на клиента» внутри более дробного графика.
:::

## Зачем это нужно

Обычная мера считается на том уровне детализации, что на листе. Но часто нужен агрегат на **другом** уровне: например, показать заказы по дням, но рядом — итог по клиенту целиком. LOD решает это, не ломая визуализацию.

## Зачем именно LOD

Без LOD пришлось бы делать отдельный лист или хитрые table calc. LOD считает «вбок» на нужном уровне прямо в одном поле:

- «Сколько всего потратил клиент» — независимо от текущей разбивки по датам.
- «Первая дата заказа клиента» — для когорт.
- «Средняя выручка на клиента», а не на строку.

## FIXED

`FIXED` фиксирует измерения, по которым считать, **игнорируя фильтры разбивки** на листе:

```
{ FIXED [Customer ID] : SUM([Sales]) }
```

→ для каждой строки подставит общую сумму её клиента, даже если на листе разбивка по дням. Дальше можно посчитать долю:

```
SUM([Sales]) / ATTR({ FIXED [Customer ID] : SUM([Sales]) })
```

## INCLUDE и EXCLUDE

В отличие от `FIXED`, они работают **относительно** текущего уровня листа:

- **`INCLUDE`** — добавить измерение к расчёту, даже если его нет на полках. Пример: средний чек на клиента, показанный в разрезе региона — `{ INCLUDE [Customer ID] : SUM([Sales]) }`, затем усреднить.
- **`EXCLUDE`** — убрать измерение из расчёта. Пример: доля категории внутри общей суммы региона, исключив саму категорию.

| Тип | Уровень расчёта |
|-----|-----------------|
| `FIXED` | строго по указанным измерениям (игнорирует лист) |
| `INCLUDE` | уровень листа + указанные измерения |
| `EXCLUDE` | уровень листа − указанные измерения |

:::caution[FIXED и фильтры: порядок имеет значение]
`FIXED` вычисляется **до** обычных фильтров измерений (но после context-фильтров). Поэтому фильтр на дашборде может не повлиять на `FIXED`-расчёт так, как ждёшь. Если нужно, чтобы фильтр действовал на FIXED, сделай его **context filter**.
:::

## Пример из реальной работы

Задача: на графике заказов по дням показать, какие клиенты потратили суммарно > 10 000.

```
// общая сумма клиента, независимо от дробления по дням
{ FIXED [Customer ID] : SUM([Sales]) } > 10000
```

Это поле-флаг можно бросить в фильтр — и на дневном графике останутся только «крупные» клиенты целиком.

<details>
<summary>1. На графике продажи по дням, а нужна доля каждого дня в общей выручке клиента. Чем?</summary>

LOD `FIXED` по клиенту: `{ FIXED [Customer ID] : SUM([Sales]) }` даёт сумму клиента независимо от дневной разбивки. Поделив дневную сумму на неё, получишь долю дня. Обычная мера так не умеет — она считается на уровне листа (день).

</details>

<details>
<summary>2. Чем FIXED отличается от INCLUDE?</summary>

`FIXED` считает строго по перечисленным измерениям, игнорируя то, что на полках листа. `INCLUDE` берёт уровень детализации листа и **добавляет** к нему указанные измерения. То есть FIXED абсолютен, INCLUDE/EXCLUDE — относительны текущей визуализации.

</details>
