:::tip[Коротко]
Amplitude — популярная платформа продуктовой аналитики на основе [событий](/08-product-analytics/07-event-tracking/). Без SQL, мышкой строишь **сегментацию**, **воронки** и **retention**, собираешь **когорты** (в т.ч. поведенческие) и дашборды. Качество всего этого упирается в качество логирования событий.
:::

:::note[Поток данных]
Вход: продукт шлёт события через SDK/HTTP API (по [tracking plan](/08-product-analytics/07-event-tracking/))
→ Обработка: Amplitude склеивает события с пользователями и считает метрики кликами в UI
→ Выход: воронки, retention, сегментация, когорты, дашборды.
Зачем: продакт и аналитик считают продуктовые метрики сами, без выгрузок из БД.
:::

## Зачем это нужно

Amplitude/Mixpanel часто требуют в продуктовых вакансиях: они дают считать метрики самостоятельно, без обращения к инженерам и SQL. Знание Amplitude — частый плюс в резюме продуктового аналитика.

## Как данные попадают внутрь

Amplitude — облачный сервис (SaaS): регистрируешь проект, получаешь **API key**, и продукт начинает слать события:

- **SDK** — для web, iOS/Android, бэкенда; внедряют разработчики по [tracking plan](/08-product-analytics/07-event-tracking/).
- **Server-to-server HTTP API** — события шлёт бэкенд напрямую (для серверных действий вроде оплаты).
- **Импорт/коннекторы** — историческая загрузка, интеграции с CDP (Segment) и [DWH](/11-modern-stack/01-cloud-dwh-overview/).

Аналитик обычно SDK не пишет, но **проектирует, какие события и свойства логировать** — и от этого зависит вся последующая аналитика.

## Как устроены данные

Три кирпича (подробно — [event tracking](/08-product-analytics/07-event-tracking/)):

- **Event** — действие (`order_completed`) с **event properties** (`amount`, `payment_method`).
- **User properties** — атрибуты пользователя (`plan`, `country`).
- **Identity** — Amplitude склеивает анонимные действия (`device_id`) с залогиненным `user_id`, чтобы путь пользователя был единым до и после регистрации.

## Как работать: основные отчёты

| Отчёт | Отвечает на вопрос |
|-------|--------------------|
| **Segmentation** | сколько раз происходит событие, в разрезе свойств |
| **Funnel** | где отваливаются в последовательности шагов |
| **Retention** | возвращаются ли пользователи со временем |
| **User paths / Flows** | какими путями ходят по продукту |

Это те же [воронки](/08-product-analytics/03-funnels/) и [retention](/08-product-analytics/05-retention-curves/), но кликами поверх событий. Базовый отчёт — **Event Segmentation**: выбираешь событие (`order_completed`), метрику (число событий / уникальные пользователи) и разрез по свойству (`country`, `platform`).

**Cohorts:**
- **Cohort** — сохранённая группа (например, «зарегистрировались в марте»).
- **Behavioral cohort** — группа по **поведению** («сделали ≥ 3 покупки», «использовали фичу X, но не Y»). Очень мощно: подставляешь такую когорту фильтром в любой отчёт.

Несколько отчётов собираются в **дашборд** для регулярного мониторинга.

:::note[Amplitude считает по уникальным пользователям]
По умолчанию многие отчёты оперируют уникальными пользователями, а не событиями — это правильно для продуктовых метрик (DAU, retention). Но всегда проверяй, что выбрано в метрике («unique users» vs «event totals»): спутаешь — получишь несопоставимые цифры.
:::

:::caution[Границы: Amplitude — не DWH]
Amplitude силён в готовых продуктовых отчётах по событиям, но это **не замена SQL и хранилища**: произвольный JOIN с биллингом/CRM, сложная кастомная логика и «сырой» доступ к данным ограничены. Для глубокого ad-hoc сырые события часто **экспортируют в [DWH](/11-modern-stack/01-cloud-dwh-overview/)** и считают там. Amplitude — для быстрых продуктовых вопросов, не для всей аналитики компании.
:::

**Итог потока:** продукт логирует события → Amplitude склеивает их с пользователями → ты кликами получаешь воронки, retention и сегменты, не написав ни строчки SQL.

<details>
<summary>1. Нужно найти, на каком шаге онбординга отваливаются пользователи. Какой отчёт?</summary>

Funnel (воронка): задаёшь последовательность событий онбординга, и Amplitude покажет конверсию и dropoff на каждом шаге. Дальше сегментируешь по платформе/каналу, чтобы понять, у кого именно провал. Это [funnel-анализ](/08-product-analytics/03-funnels/) без SQL.

</details>

<details>
<summary>2. Чем поведенческая когорта полезнее когорты по дате регистрации?</summary>

Она группирует по действиям («сделали 3+ покупки», «активировали фичу»), а не по времени прихода. Это позволяет изучать и таргетировать пользователей по реальному поведению — например, сравнить retention тех, кто дошёл до ключевого действия, с теми, кто нет. Гибче, чем фиксированная дата.

</details>

<details>
<summary>3. Продакт просит сложный отчёт с JOIN событий и данных биллинга из CRM. Делать в Amplitude?</summary>

Скорее нет: Amplitude заточен под события, произвольные джойны с внешними системами и сырой доступ в нём ограничены. Такой анализ удобнее на выгрузке событий в DWH + SQL. Amplitude — для быстрых продуктовых вопросов (воронки, retention), а не для кросс-системной аналитики.

</details>
