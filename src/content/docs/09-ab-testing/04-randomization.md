:::tip[Коротко]
Рандомизация — сердце A/B-теста. Ключевое решение — **юнит рандомизации**: чаще всего пользователь (а не сессия), иначе один человек попадёт в обе группы и результат сломается. Деление делают **детерминированным хешированием** ID, а баланс групп проверяют до и после.
:::

## Зачем это нужно

Именно случайное деление делает группы сопоставимыми и [превращает корреляцию в причинность](/09-ab-testing/01-fundamentals/). Ошибка в рандомизации (не тот юнит, перекос групп) обесценивает весь тест, как бы аккуратно потом ни считали статистику.

## Чем рандомизировать (юнит)

| Юнит | Когда |
|------|-------|
| **User** | по умолчанию — один опыт на человека через все сессии |
| **Session** | если эффект внутри сессии и переключение не путает |
| **Device / cookie** | когда нет логина (анонимы) |
| **Cluster** (город, команда) | при [сетевых эффектах](/09-ab-testing/01-fundamentals/) |

:::caution[Рандомизация по сессии часто ломает тест]
Если делить по сессиям, один пользователь в понедельник увидит вариант A, во вторник — B. Он «загрязняет» обе группы и путается сам (то старый, то новый интерфейс). Для большинства продуктовых тестов юнит — **пользователь**: один человек = один стабильный вариант на всё время теста.
:::

## Хеширование для воспроизводимости

Группу определяют **детерминированно**: `hash(user_id + test_name) % 100` → стабильное число 0–99, по которому юзер всегда попадает в ту же группу.

- **Воспроизводимо**: один и тот же пользователь всегда в одной группе (не «прыгает» между визитами).
- **Соль из имени теста** (`+ test_name`): в разных тестах деление независимое, а не одни и те же люди вечно в «B».

## Стратификация

Чтобы важные сегменты разделились поровну, делят **внутри страт** (платформа, страна, новый/старый): сначала разбиваем на страты, потом рандомизируем в каждой. Это снижает дисперсию и убирает риск, что, скажем, все iOS-юзеры случайно окажутся в одной группе. Связано с [выборками в статистике](/05-statistics/04-clt-and-sampling/).

## Контроль баланса

- **До теста (A/A-тест)** — пустить две одинаковые группы без изменения: если метрика уже «значимо» различается, рандомизация или метрика сломаны.
- **Во время/после** — проверить, что доли групп соответствуют замыслу (50/50). Резкое расхождение — это [SRM](/09-ab-testing/07-common-pitfalls/), сигнал баги в сплите.
- Сверить, что распределение ключевых атрибутов (платформа, гео) похоже в группах.

## Задачи для самопроверки

<details>
<summary>1. Почему обычно рандомизируют по пользователю, а не по сессии?</summary>

Чтобы один человек получал один и тот же вариант во всех визитах. При делении по сессии пользователь попадает то в A, то в B — загрязняет обе группы и видит скачущий интерфейс, что искажает поведение и результат. Юнит «пользователь» даёт чистый и стабильный опыт.

</details>

<details>
<summary>2. Зачем добавлять имя теста в хеш при определении группы?</summary>

Чтобы деление в разных тестах было независимым. Если хешировать только `user_id`, одни и те же пользователи всегда будут в «группе B» во всех экспериментах, накапливая систематический сдвиг. Соль из имени теста делает сплит каждого теста самостоятельным.

</details>
