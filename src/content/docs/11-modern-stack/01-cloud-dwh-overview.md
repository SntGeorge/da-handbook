:::tip[Коротко]
DWH (хранилище данных) — это база, заточенная под **аналитику** (OLAP), а не под транзакции приложения (OLTP). Главное отличие — **колоночное хранение**: данные лежат по столбцам, поэтому агрегаты по миллиардам строк считаются быстро. Облачные DWH (Snowflake, BigQuery) разделяют хранение и вычисления — платишь за то, что считаешь.
:::

## Зачем это нужно

В стартапе и крупной компании аналитик работает не с локальным PostgreSQL, а с облачным DWH, куда стекаются все данные. Понимать, чем оно отличается от обычной БД и почему запрос «дешевле» написать иначе — навык уровня middle+.

## OLTP vs OLAP

Два мира баз данных под разные задачи:

| | OLTP | OLAP (DWH) |
|--|------|------------|
| Для чего | работа приложения | аналитика |
| Операции | много мелких INSERT/UPDATE | тяжёлые SELECT с агрегатами |
| Пример запроса | «создать заказ #123» | «выручка по странам за год» |
| Хранение | строковое | колоночное |
| Пример | PostgreSQL, MySQL | Snowflake, BigQuery, ClickHouse |

Бизнес-приложение пишет в OLTP, ночью/в реальном времени данные перетекают в OLAP-хранилище, где их анализируют — чтобы тяжёлые отчёты не тормозили прод.

## Что такое DWH

Data Warehouse — централизованное хранилище, куда сводят данные из разных источников (продукт, маркетинг, финансы) в единую модель для анализа. Это «единый источник правды» компании.

## Колоночное хранение

:::tip[Почему DWH быстрый на аналитике]
В строковой БД строка лежит целиком подряд — удобно достать один заказ. В колоночной значения **одного столбца** лежат вместе. Для `SUM(amount)` по миллиарду строк колоночная читает только столбец `amount`, а не все поля, — на порядки меньше данных с диска. Плюс однотипные данные в столбце отлично сжимаются. Отсюда скорость аналитики.
:::

## Современные игроки

| Хранилище | Облако | Модель оплаты | Когда выбирают |
|-----------|--------|---------------|----------------|
| **Snowflake** | AWS/Azure/GCP | за время работы compute (кредиты) | мультиоблако, удобство, разделение нагрузок команд |
| **BigQuery** | Google Cloud | за объём просканированных данных | экосистема Google (GA4), serverless без админства |
| **Redshift** | AWS | за выделенные ноды (или serverless) | всё уже в AWS |
| **Databricks** | AWS/Azure/GCP | за compute (DBU) | тяжёлый ML/Spark, Lakehouse |
| **ClickHouse** | self-host / Cloud | свой сервер или облако | сверхскорость на событиях, популярен в СНГ |

[ClickHouse](/11-modern-stack/04-clickhouse/) в СНГ — почти стандарт для продуктовой аналитики, поэтому ему посвящена отдельная страница.

## ETL vs ELT

Важный сдвиг, на котором стоит весь современный стек. Раньше данные **трансформировали до загрузки** в хранилище (ETL), теперь — **после** (ELT):

| | ETL (старый подход) | ELT (современный) |
|--|---------------------|-------------------|
| Порядок | Extract → **Transform** → Load | Extract → Load → **Transform** |
| Где трансформация | на отдельном сервере до DWH | **внутри DWH** силами SQL |
| Инструменты | дорогие ETL-комбайны | загрузка (Fivetran/Airbyte) + [dbt](/11-modern-stack/05-dbt-basics/) |

Почему перешли на ELT: облачные DWH дёшевы в хранении и мощны в вычислениях — проще залить сырьё как есть и трансформировать SQL-запросами прямо в хранилище. Отсюда роль [dbt](/11-modern-stack/05-dbt-basics/) (слой T) и [оркестратора](/11-modern-stack/06-airflow-basics/).

## Lakehouse

Гибрид **data lake** (дешёвое хранилище сырых файлов) и **DWH** (структура и SQL). Форматы **Apache Iceberg** и **Delta Lake** добавляют к файлам в озере транзакции, схему и версионирование — получается «лучшее из двух миров». Тренд последних лет; для junior достаточно знать сам термин и идею.

## Задачи для самопроверки

<details>
<summary>1. Почему тяжёлые аналитические отчёты не гоняют прямо по боевой базе приложения?</summary>

Боевая база — OLTP, оптимизирована под мелкие транзакции; тяжёлый SELECT с агрегатами по всей истории нагрузит её и затормозит работу приложения. Поэтому данные переливают в OLAP-хранилище (DWH) с колоночным хранением, где такие запросы быстры и не мешают проду.

</details>

<details>
<summary>2. Почему колоночное хранилище быстрее считает `SUM(amount)` по миллиарду строк?</summary>

Оно читает с диска только столбец `amount`, а не целые строки со всеми полями — на порядки меньше данных. Плюс однотипные значения столбца сильно сжимаются. Строковая БД ради одной суммы вынуждена пройти все поля каждой строки.

</details>
