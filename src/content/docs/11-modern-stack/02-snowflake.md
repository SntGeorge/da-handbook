:::tip[Коротко]
Главная идея Snowflake — **разделение хранения и вычислений**: данные лежат отдельно, а считают их независимые **virtual warehouses**, которые масштабируются и платятся отдельно. Это даёт гибкость (несколько команд не мешают друг другу) и контроль затрат. Плюс уникальные фишки — **Time Travel** и **Zero-Copy Cloning**.
:::

:::note[Поток данных]
Вход: данные из источников (загрузка через `COPY`/Snowpipe/коннекторы)
→ Обработка: virtual warehouse выполняет SQL и трансформации, storage хранит данные отдельно
→ Выход: таблицы и витрины для BI и аналитики.
Зачем: масштабируемое облачное хранилище, где платишь только за использованный compute.
:::

## Зачем это нужно

Snowflake — одно из самых распространённых облачных DWH, особенно в США и Европе. Аналитику важно понимать модель затрат (за что списываются «кредиты») и пару фирменных возможностей — их любят спрашивать.

## Архитектура: storage / compute separation

В классической БД хранение и вычисления связаны. Snowflake их разделяет на три слоя:

- **Storage** — данные в облаке, платишь за объём.
- **Compute** — virtual warehouses, которые выполняют запросы; платишь за время их работы.
- **Cloud services** — оптимизатор, метаданные, безопасность.

Следствие: можно остановить вычисления (warehouse «спит» — не платишь), а данные останутся; и наоборот — нарастить мощность под тяжёлый запрос, не трогая хранилище.

## Virtual warehouses

«Warehouse» в Snowflake — это **вычислительный кластер**, а не хранилище (сбивает с толку новичков). Размеры от XS до 6XL:

- Каждая команда/нагрузка может иметь свой warehouse — не мешают друг другу.
- Можно настроить **auto-suspend** (засыпать без запросов) и **auto-resume**.

```sql
-- создаём вычислительный кластер с авто-засыпанием через 60 сек простоя
CREATE WAREHOUSE analytics_wh
  WAREHOUSE_SIZE = 'XSMALL'   -- XS, S, M, L, XL ... 6XL
  AUTO_SUSPEND = 60           -- секунд простоя до сна (не жжём кредиты впустую)
  AUTO_RESUME = TRUE;         -- сам проснётся на следующем запросе
```

:::note[Размер warehouse и кредиты]
Каждый шаг размера **удваивает** и мощность, и расход: XS = 1 кредит/час, S = 2, M = 4, L = 8 и т.д. Кредит тарифицируется посекундно (минимум 60 сек после пробуждения). Вывод: для дашбордов и ad-hoc хватает XS/S; большой warehouse берут только под реально тяжёлую трансформацию — и обязательно с `AUTO_SUSPEND`.
:::

## Time Travel и Zero-Copy Cloning

Две фирменные возможности:

- **Time Travel** — запросить данные «как было N дней назад» или восстановить удалённое:

```sql
-- состояние таблицы 1 час назад (до кривого UPDATE)
SELECT * FROM orders AT (OFFSET => -3600);
-- вернуть случайно дропнутую таблицу одной командой
UNDROP TABLE orders;
```

Окно хранения по умолчанию — 1 день (на Enterprise до 90).

- **Zero-Copy Cloning** — мгновенная копия без дублирования данных (копируются метаданные, физически данные общие до изменений):

```sql
-- полная копия прод-базы для тестов за секунды и без затрат на хранение
CREATE DATABASE analytics_dev CLONE analytics_prod;
```

## Особенности SQL

Snowflake SQL близок к стандарту, плюс полезные расширения:

```sql
-- VARIANT хранит JSON как есть; обращаемся через : и разворачиваем FLATTEN
SELECT t.value:item_id::int AS item_id
FROM orders, LATERAL FLATTEN(input => orders.items) t;

-- QUALIFY фильтрует по оконной функции без обёртки-подзапроса
SELECT order_id, amount,
       ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY amount DESC) AS rn
FROM orders
QUALIFY rn = 1;     -- топ-1 заказ на клиента
```

`VARIANT` + `FLATTEN` — для полуструктурированных данных (JSON из API), `QUALIFY` — частый аналог фильтра по [оконным функциям](/02-sql/09-window-functions/).

## Кредиты и оптимизация затрат

:::caution[В Snowflake платят за время работы compute]
Биллинг идёт в **кредитах** за время работы warehouse (не за число запросов). Забытый большой warehouse, который не «уснул», жжёт деньги впустую. Базовая оптимизация: включай auto-suspend, бери размер под задачу (не 4XL ради мелкого запроса), не сканируй лишнее. Аналитик, пишущий неэффективные запросы на огромном warehouse, обходится компании дорого.
:::

## Задачи для самопроверки

<details>
<summary>1. «Warehouse» в Snowflake — это где хранятся данные?</summary>

Нет, это распространённая путаница. Virtual warehouse в Snowflake — вычислительный кластер (compute), который выполняет запросы. Данные хранятся отдельно, в слое storage. Разделение хранения и вычислений — ключевая идея архитектуры.

</details>

<details>
<summary>2. Случайно удалил данные из таблицы. Что в Snowflake поможет?</summary>

Time Travel: можно запросить состояние таблицы до удаления или восстановить её в пределах окна хранения (по умолчанию до нескольких дней). Это фирменная возможность Snowflake, спасающая после ошибочных DELETE/DROP без долгого восстановления из бэкапа.

</details>
