:::tip[Коротко]
ClickHouse — сверхбыстрая колоночная OLAP-БД с открытым кодом, созданная в Яндексе. Заточена под **аналитические запросы в масштабе** (миллиарды строк за секунды). Сердце — семейство движков **MergeTree**. В СНГ-вакансиях встречается очень часто, поэтому знать её основы особенно полезно именно здесь.
:::

## Зачем это нужно

В СНГ ClickHouse — почти стандарт для продуктовой аналитики (его используют Яндекс и множество компаний вокруг). Если ищешь работу здесь, велик шанс работать именно с ним, а не со Snowflake/BigQuery. Это [отмечено в обзоре стека](/00-intro/market-stack-2026/).

## Зачем ClickHouse: OLAP в масштабе

ClickHouse невероятно быстр на агрегатах по огромным данным за счёт колоночного хранения, сжатия и векторизованного выполнения. Типичный сценарий — аналитика событий: миллиарды строк логов/кликов, по которым нужны быстрые `GROUP BY`.

:::caution[ClickHouse — не для частых UPDATE/DELETE]
Это OLAP-движок: он блестящ на чтении и вставке пачками, но **точечные UPDATE/DELETE дорогие и асинхронные** (через `ALTER ... UPDATE`, «мутации»). Не используй ClickHouse как обычную транзакционную БД приложения — для этого есть [OLTP](/11-modern-stack/01-cloud-dwh-overview/). Он для аналитики, а не для «обновить статус заказа».
:::

## MergeTree engine

Основное семейство таблиц. Идея: данные пишутся частями (parts), которые фоном сливаются (merge), и хранятся отсортированными по **ключу сортировки** (`ORDER BY` в DDL):

```sql
CREATE TABLE events (
    event_date Date,
    user_id UInt64,
    event_type String,
    amount Float64
) ENGINE = MergeTree()
ORDER BY (event_date, user_id);     -- ключ сортировки = быстрые диапазонные запросы
```

Сортировка по ключу даёт быстрый поиск по диапазону и работает как первичный индекс. Часто добавляют партиционирование по месяцу (`PARTITION BY toYYYYMM(event_date)`).

У MergeTree есть **варианты движка** под разные задачи — они по-разному «схлопывают» строки с одинаковым ключом сортировки при фоновом слиянии:

| Движок | Что делает при слиянии | Когда |
|--------|------------------------|-------|
| **MergeTree** | ничего, хранит все строки | сырые события |
| **ReplacingMergeTree** | оставляет последнюю версию по ключу | дедупликация, «последнее состояние» |
| **SummingMergeTree** | суммирует числовые столбцы по ключу | предагрегаты |
| **AggregatingMergeTree** | хранит состояния агрегатов | сложные предрасчёты с MV |

:::caution[«Схлопывание» происходит не сразу]
ReplacingMergeTree убирает дубли только при фоновом слиянии частей — момент не контролируется. Чтобы гарантированно увидеть схлопнутый результат в запросе, добавляй модификатор `FINAL` (`SELECT ... FROM t FINAL`) — но он медленный, не злоупотребляй им на больших данных.
:::

## ARRAY JOIN

ClickHouse умеет хранить **массивы** в ячейке (например, список товаров в заказе). `ARRAY JOIN` «разворачивает» массив в строки — аналог unnest:

```sql
SELECT user_id, item
FROM orders
ARRAY JOIN items AS item;     -- одна строка с [a,b,c] → три строки
```

Удобно для событийных данных, где атрибуты приходят списками.

## Материализованные представления

В ClickHouse **materialized view** работает как триггер на вставку: при записи в исходную таблицу автоматически считает агрегат и складывает в другую таблицу. Так заранее считают «выручку по дням» на лету, и дашборды читают готовый маленький результат вместо пересчёта по миллиардам строк.

```sql
-- MV считает дневную выручку прямо при вставке новых событий
CREATE MATERIALIZED VIEW revenue_daily_mv
ENGINE = SummingMergeTree()
ORDER BY day AS
SELECT toDate(event_date) AS day, sum(amount) AS revenue
FROM events
GROUP BY day;
```

Теперь дашборд читает крошечную `revenue_daily_mv` (одна строка на день) вместо агрегации миллиардов строк `events` при каждом открытии.

## Задачи для самопроверки

<details>
<summary>1. Команда хочет хранить в ClickHouse заказы с частым обновлением статуса. Хорошая идея?</summary>

Плохая. ClickHouse — OLAP-движок, точечные UPDATE дорогие и асинхронные (мутации). Частое обновление статусов — это OLTP-нагрузка, для неё нужна транзакционная БД (PostgreSQL). ClickHouse держи для аналитики событий: быстрые чтения и пакетные вставки, а не построчные апдейты.

</details>

<details>
<summary>2. Зачем в ClickHouse материализованное представление?</summary>

Чтобы считать агрегаты заранее, в момент вставки данных. MV работает как триггер: новые строки в исходной таблице автоматически агрегируются в целевую (например, выручка по дням). Дашборды затем читают маленький предрассчитанный результат вместо тяжёлого пересчёта по миллиардам строк при каждом запросе.

</details>
