:::tip[Коротко]
dbt превращает трансформации в DWH в **код, как у разработчиков**: модель — это `SELECT` в `.sql`-файле, а dbt строит из них таблицы/представления, разруливает зависимости, тестирует и документирует. Это слой **T (Transform)** в ELT: сырые данные уже в хранилище, dbt делает из них чистые витрины. Один из главных навыков analytics engineer.
:::

:::note[Поток данных]
Вход: сырые таблицы в DWH (объявлены как `sources`)
→ Обработка: SQL-модели по слоям (staging → marts), связаны через `ref()`, прогоняются `dbt build` с тестами
→ Выход: чистые витрины + документация и граф зависимостей (lineage).
Зачем: трансформации как версионируемый, тестируемый, воспроизводимый код.
:::

## Зачем это нужно

Раньше трансформации жили в разрозненных SQL-скриптах и вьюхах без версий, тестов и порядка запуска — хаос. dbt приносит в аналитику инженерные практики: git, тесты, документацию, зависимости. Всё чаще встречается в вакансиях middle+.

## Что такое dbt и зачем

dbt (data build tool) отвечает за **T** в **ELT**: данные сначала грузят сырыми в [DWH](/11-modern-stack/01-cloud-dwh-overview/) (E и L), а потом dbt трансформирует их **внутри** хранилища SQL-запросами. Никаких отдельных серверов обработки — вся мощь считает сам DWH.

## Как поставить и подключить к DWH

dbt Core — это Python-пакет; ставится под конкретный склад (адаптер):

```bash
pip install dbt-postgres        # или dbt-snowflake, dbt-bigquery, dbt-clickhouse
dbt init my_project             # создаёт каркас проекта
```

Куда подключаться — в `profiles.yml` (хост, база, схема, креды):

```yaml
my_project:
  target: dev
  outputs:
    dev:
      type: postgres
      host: localhost
      schema: analytics
      user: "{{ env_var('DB_USER') }}"   # креды — из переменных окружения, не в файл
```

:::caution[Креды — в переменные окружения, не в git]
`profiles.yml` с паролем не коммить. Бери секреты через `env_var(...)` из переменных окружения — иначе доступ к складу утечёт в репозиторий.
:::

## Models, sources, seeds

- **Model** — `.sql`-файл с `SELECT`. dbt оборачивает его в `CREATE TABLE/VIEW` сам. Модели ссылаются друг на друга через `ref()`, и dbt строит граф зависимостей.
- **Source** — описание сырых исходных таблиц (`source()`), чтобы ссылаться на них явно и тестировать.
- **Seed** — небольшой CSV (справочники), загружаемый в DWH командой `dbt seed`.

```sql
-- models/marts/revenue_by_country.sql
SELECT country, SUM(amount) AS revenue
FROM {{ ref('stg_orders') }}      -- ссылка на другую модель
WHERE status = 'paid'
GROUP BY country
```

`ref()` — сердце dbt: связывает модели, и `dbt run` строит их в правильном порядке.

## Структура проекта и слои

Типичный проект разбит на **слои**, и это общепринятая конвенция, которую спрашивают:

```text
models/
  staging/    # stg_*  — 1:1 с источником: переименовать, привести типы
  marts/      # fct_*, dim_* — бизнес-витрины, звезда (см. моделирование)
dbt_project.yml   # конфиг проекта
```

- **staging** — лёгкая очистка сырых источников (один stg-файл на одну таблицу источника), материализуется как `view`.
- **marts** — финальные витрины для аналитики и BI, материализуются как `table`/`incremental`.

Поток: `source → staging → marts`, связи — через `ref()`/`source()`.

## Основные команды

```bash
dbt run            # построить все модели (CREATE TABLE/VIEW)
dbt test           # прогнать тесты данных
dbt build          # run + test + seed по графу зависимостей (рекомендуется)
dbt docs generate  # собрать документацию и граф зависимостей (lineage)
```

## Материализации

Как именно модель «материализуется» в DWH:

| Тип | Что создаёт | Когда |
|-----|-------------|-------|
| **view** | представление | лёгкие модели, всегда свежо |
| **table** | физическая таблица | тяжёлые расчёты, читают часто |
| **incremental** | дописывает только новые строки | большие растущие таблицы |
| **ephemeral** | подставляется как CTE, не материализуется | промежуточная логика |

`incremental` критичен для больших данных: пересчитывать миллиарды строк каждый раз дорого. Внутри incremental-модели через `is_incremental()` дописывают только свежее:

```sql
{{ config(materialized='incremental', unique_key='order_id') }}

SELECT * FROM {{ ref('stg_orders') }}
{% if is_incremental() %}
  -- при инкрементальном прогоне берём только новее, чем уже загружено
  WHERE updated_at > (SELECT MAX(updated_at) FROM {{ this }})
{% endif %}
```

Первый `dbt run` строит таблицу целиком, последующие — дописывают только новые строки.

## Тесты

dbt из коробки умеет тестировать данные декларативно (в YAML): `unique`, `not_null`, `accepted_values`, `relationships` (ссылочная целостность). Подробнее — в [качестве данных](/11-modern-stack/08-data-quality/).

```yaml
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests: [unique, not_null]
```

## Jinja и macros

dbt-модели — это SQL + шаблонизатор **Jinja**: можно подставлять переменные, циклы, условия и выносить повторяющуюся логику в **macros** (аналог функций). Так избегают копипасты SQL. На старте достаточно понимать `ref()`, `source()` и базовые `{{ }}`.

## dbt Cloud vs Core

- **dbt Core** — бесплатный open-source CLI, запускаешь сам.
- **dbt Cloud** — платный сервис: веб-IDE, планировщик, хостинг документации.

:::note[dbt не хранит и не грузит данные]
Частое заблуждение: dbt — это не база и не загрузчик. Он только **трансформирует** уже загруженные в DWH данные, генерируя и выполняя SQL **в твоём хранилище**. За загрузку (E/L) отвечают другие инструменты (Fivetran, Airbyte), за оркестрацию — [Airflow](/11-modern-stack/06-airflow-basics/).
:::

## Задачи для самопроверки

<details>
<summary>1. Зачем в dbt-моделях писать `{{ ref('other_model') }}` вместо имени таблицы напрямую?</summary>

`ref()` сообщает dbt о зависимости между моделями: он строит граф и запускает их в правильном порядке, а также подставляет корректное имя таблицы в нужной схеме/окружении. Хардкод имени ломает порядок сборки и переносимость между dev/prod. `ref()` — основа управления зависимостями в dbt.

</details>

<details>
<summary>2. Таблица событий растёт на миллионы строк в день, модель пересчитывается долго. Какая материализация?</summary>

`incremental`: dbt будет дописывать только новые/изменённые строки, а не пересчитывать всю историю каждый запуск. Для больших растущих таблиц это резко экономит время и деньги DWH. `table` пересчитывал бы всё заново, `view` — считал бы на каждом чтении.

</details>
