:::tip[Коротко]
Качество данных — это автоматические проверки, что данные не «протухли»: ключи уникальны, нет неожиданных `NULL`, значения в допустимом диапазоне. Базовые тесты встроены в [dbt](/11-modern-stack/05-dbt-basics/) (`unique`, `not_null`, `accepted_values`), глубже — Great Expectations, а непрерывный мониторинг — data observability. Без этого «дашборд врёт, а никто не знает».
:::

## Зачем это нужно

Главный кошмар аналитика — принять решение по сломанным данным (задвоились заказы, источник прислал мусор, метрика молча обнулилась). Тесты данных ловят это **автоматически**, до того как цифры попадут в отчёт руководству. Это часть инженерной зрелости аналитики.

## Что такое data quality

Набор проверок и процессов, гарантирующих, что данные **корректны, полны и свежи**. Проверяют по нескольким измерениям: точность, полнота, уникальность, актуальность (freshness), согласованность.

## Виды тестов

| Тест | Что проверяет |
|------|---------------|
| **not_null** | в столбце нет пропусков (например, `order_id`) |
| **unique** | значения уникальны (нет задвоения ключа) |
| **accepted_values** | значение из допустимого набора (`status ∈ {paid, cancelled}`) |
| **relationships** | ссылочная целостность (каждый `customer_id` есть в справочнике) |
| **custom** | своё бизнес-правило (`amount >= 0`, выручка не упала на 90% за день) |

`unique` + `not_null` на первичном ключе ловят самую частую беду — [задвоение строк при джойне](/02-sql/06-joins/).

## Тесты dbt

dbt делает тесты декларативными — описываешь в YAML, запускаешь `dbt test`:

```yaml
models:
  - name: orders
    columns:
      - name: order_id
        tests: [unique, not_null]
      - name: status
        tests:
          - accepted_values:
              values: ['paid', 'cancelled', 'pending']
```

Тесты гоняются вместе с пайплайном ([Airflow](/11-modern-stack/06-airflow-basics/)): данные не проходят проверку — пайплайн падает, и битые данные не доезжают до дашборда.

**Свежесть (freshness)** — отдельный важный тест: «данные не старше N часов». Описывается у источника:

```yaml
sources:
  - name: shop
    tables:
      - name: orders
        loaded_at_field: _loaded_at
        freshness:
          warn_after:  { count: 12, period: hour }   # предупредить
          error_after: { count: 24, period: hour }   # уронить пайплайн
```

Запускается `dbt source freshness`. Это ловит самый коварный случай — тесты зелёные, но **данные просто не приехали**.

**Кастомный тест** (singular) — это обычный SQL-файл в `tests/`, который должен вернуть **ноль строк**; вернул строки — тест провален:

```sql
-- tests/assert_amount_non_negative.sql — ищем «плохие» строки
SELECT order_id FROM {{ ref('orders') }} WHERE amount < 0
```

:::note[Severity: warn vs error]
У теста есть уровень: `error` (по умолчанию — роняет пайплайн) и `warn` (только предупреждает). Критичное (уникальность ключа) — `error`; «подозрительное, но не смертельное» (резкий скачок объёма) — часто `warn`, чтобы не блокировать всю загрузку из-за мелочи.
:::

## Great Expectations

Отдельная Python-библиотека для более глубоких и гибких проверок («ожиданий»): распределение значений, доля null, статистические профили, подробные отчёты о валидации. Берут, когда встроенных dbt-тестов мало.

## Data observability

:::tip[Observability — мониторинг данных, как APM для серверов]
Платформы вроде **Monte Carlo**, Soda, Bigeye следят за данными непрерывно и **автоматически** сигналят об аномалиях: таблица не обновилась вовремя (freshness), объём строк резко скакнул, схема изменилась, распределение поехало. В отличие от заранее прописанных тестов, observability ловит и то, чего ты не предусмотрел. Зрелый уровень data quality.
:::

<details>
<summary>1. Какие два теста на первичном ключе ловят самую частую проблему джойнов?</summary>

`unique` и `not_null` на ключе (например, `order_id`). Они гарантируют, что ключ уникален и заполнен, — это предотвращает задвоение строк при джойнах и «потерянные» записи. Та же ловушка fan-out из SQL: дубль в ключе раздувает суммы.

</details>

<details>
<summary>2. Тесты dbt все зелёные, но дашборд за вчера пустой. Что могло не сработать и что поможет?</summary>

Тесты проверяют то, что заложили, но могли не покрыть **freshness** — данные просто не приехали (источник не отдал, пайплайн не запустился). Поможет тест на свежесть и data observability (Monte Carlo и т.п.), которые следят за обновляемостью и аномалиями автоматически, в том числе за непредусмотренными случаями.

</details>
