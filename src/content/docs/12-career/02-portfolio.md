:::tip[Коротко]
Без коммерческого опыта портфолио **заменяет** строчку «опыт работы»: оно доказывает, что ты реально умеешь, а не просто прошёл курсы. Минимум — 1–2 законченных проекта на **GitHub** с понятным README и, если есть дашборды, ссылка на **Tableau Public / Looker Studio**. Один доведённый до конца проект сильнее десяти заброшенных.
:::

## Зачем это нужно

«Прошёл курс по SQL» — слова. Проект с кодом, дашбордом и выводами — доказательство. Для джуна без опыта портфолио часто решает, позовут ли на собеседование. Это твой единственный способ показать навык до интервью.

## GitHub

Главная витрина кода. На что смотрит рекрутер:

- **README проекта** — самое важное: что за проект, какой вопрос решал, какие данные, что получилось (с графиками-скриншотами). README читают чаще, чем код.
- **Структура** — аккуратные папки (`data/`, `notebooks/`, `README.md`), а не свалка файлов.
- **Чистые ноутбуки** — с markdown-пояснениями и выводами, прогоняемые сверху вниз ([как в Jupyter](/04-python/02-jupyter-workflow/)).

:::caution[Не коммить данные и секреты]
В репозиторий не должны попадать большие датасеты, пароли, API-ключи, `.env`. Добавь `.gitignore` ([см. Git](/01-setup/git-github/)). Закоммиченный ключ — и плохо для безопасности, и плохой сигнал рекрутеру о твоей аккуратности.
:::

## Tableau Public / Looker Studio

Для дашбордов код в GitHub не нагляден — нужна **живая интерактивная ссылка**. [Tableau Public](/07-bi-tools/tableau/07-publishing/) и [Looker Studio](/07-bi-tools/looker/01-intro/) бесплатны и дают публичный URL, который кладёшь прямо в резюме. Рекрутер кликает и сразу видит твой дашборд.

## Kaggle

Площадка с датасетами и соревнованиями. Полезна для:

- **Notebooks** — публичные ноутбуки с анализом (видно ход мысли).
- **Datasets** — готовые данные для пет-проектов.
- Соревнования — опционально; место в топе не обязательно, но участие показывает практику.

## Статьи (Medium / Хабр)

Разбор своего проекта в статье — сильный бонус: показывает [storytelling](/06-visualization/04-storytelling/) и умение объяснять. Не обязательно, но выделяет среди джунов и сам по себе работает на личный бренд.

## Что показывать

| Формат | Для чего |
|--------|----------|
| GitHub-репозиторий | код, SQL, ноутбуки, README |
| Tableau Public / Looker | интерактивные дашборды |
| Kaggle notebook | анализ с пояснениями |
| Статья | storytelling, разбор кейса |

## Задачи для самопроверки

<details>
<summary>1. Что важнее в GitHub-проекте для рекрутера — код или README?</summary>

README. Рекрутер (и часто нанимающий менеджер) сначала читает описание: какой вопрос решал проект, на каких данных, что получилось, со скриншотами результатов. Без понятного README даже хороший код выглядит как свалка файлов. Код смотрят уже после, если README зацепил.

</details>

<details>
<summary>2. Десять начатых проектов или один законченный — что лучше в портфолио?</summary>

Один доведённый до конца: с данными, анализом, дашбордом/выводами и оформленным README. Он показывает, что ты умеешь доводить до результата — именно это нужно работодателю. Десять заброшенных полупроектов производят обратное впечатление.

</details>
